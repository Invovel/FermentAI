## 一、起点：跟风之后的自问

### 最初的困惑

做过很多项目——LLM-Wiki、Research-Skill、Agent-team，也学了 RAG 的概念。但始终有一个问题：

> **"这有什么用？"**

不是没有写代码，不是没有文档，而是**这些零件组装起来之后，解决的到底是什么问题**？

### 具体表现

- LLM-Wiki 有知识库、热值、衰减、BM25搜索，但**搜出来只是文件**，没有智能问答
- Research-Skill 能搜 arXiv、下载论文，但**下载了就放在那里**，没有再被使用
- Agent-team 有设计文档、能力路由概念，但**没跟知识库连在一起**，只是纸上谈兵
- RAG 的知识停留在"能检索不能使用"的程度，只是一个会查资料的 skill

### 跟风做项目的本质

```
看到热门词 → 实现功能 → 拼在一起 → 不知道解决什么问题
RAG      → 向量检索 → 有了
热值     → 前端显示 → 有了
衰减     → 公式写好 → 有了
BM25     → 搜索可用 → 有了
```

**结果是：零件齐全，但拼不成一辆能开的车。**

## 二、问题诊断：为什么觉得没用

### 诊断过程

通过实际分析 Obsidian Vault（`D:\Obsidian\Hub\WikiVault`）的数据，发现了关键事实：

### 2.1 关联只体现在文件夹层级

```
wiki/
├── concepts/      ← 11个概念页
├── summaries/     ← 38个摘要页
├── entities/      ← 9个实体页
└── synthesis/     ← 2个综合页
```

**问题**：
- 11个概念页中，7个正文内没有任何 `[[wikilinks]]`——它们是"信息黑洞"
- `parent_concepts` 覆盖率仅 36%，`related_concepts` 覆盖率仅 45%
- 9个实体页中只有2个被引用过——其余7个是死节点

→ **关联是死的，只有文件夹层级，没有内容级别的智能连接。**

### 2.2 热值系统完全空转

```
hot_concepts.json → {"concepts": {}}          ← 完全空对象
_hotlist.json     → {"hot_entries": []}       ← 完全空数组
_epochs.json      → heat 全部为 1.0           ← 衰减从未执行
```

**因果链**：
```
hot collect（收集）→ 标记为"待实现" → 从未运行
    ↓
hot_concepts.json → 空对象
    ↓
BM25 搜索的热词偏置 → 偏置数据为空 → 实际权重 = 0
    ↓
热值对检索的影响 = 零
```

→ **数据生产了，但没有消费者。热值写在页面上，搜索从来不参考它。**

### 2.3 Claude 接入了但无感

Claude 只在用户**主动触发**时工作（手动导入论文、手动提问），系统没有设计**主动推送**机制。热值异常检测、关联断裂修复、知识缺口发现——这些在 skill 文档里写了，但**没有自动化触发器**去驱动。

→ **AI 在那里，但它不会主动做事。**

## 三、核心洞察：不是存储，而是发酵

### 从"死数据"到"活知识"

**传统知识管理**：
```
数据 → 存储 → 查询 → 结束（数据不变）
```

**FermentAI 的答案——知识发酵**：
```
数据 → 存储 → 使用 → 产生痕迹 → 痕迹积累 → 触发规则 → 生成新关联 → 知识进化
         ↑___________________________________________________________↓
```

### 发酵的三要素

| 要素 | 说明 | 类比酿酒 |
|------|------|----------|
| **原料** | 原始知识（论文/代码/对话） | 粮食 |
| **酒曲** | 使用痕迹（查询/引用/复用） | 酒曲 |
| **时间** | 积累与衰减 | 发酵时间 |
| **产物** | 关联图谱 + 热值分布 + Agent能力画像 | 美酒 |

### 为什么"发酵"是核心

因为所有已有的技术（RAG、热值、衰减、图表）**单拿出来都没用**，只有串成一条流动的线，才有价值：

```
Research-Skill（手）→ 采集数据
    ↓
发酵引擎（酵母）→ 建立关联、积累热值、触发衰减
    ↓
LLM-Wiki（记忆）→ 存储 + 语义检索 + 关联图
    ↓
Agent-team（脑）→ 路由查询、学习模式、协同工作
```

## 四、架构跃迁：从单点到生态

### 跃迁前的状态

```
三个独立的项目：
├── LLM-Wiki      → 能搜索，不能问答，热值空转
├── Research-Skill → 能下载，不能关联，下载即遗忘
└── Agent-team    → 有设计，没连接，纸上谈兵
```

### 跃迁后的状态

```
FermentAI 统一生态：

Research-Skill ──→ 发酵引擎 ──→ LLM-Wiki ──→ Agent-team
    (搜索采集)       (关联进化)     (存储检索)     (编排调度)
         ↑                                              │
         └────────── 使用痕迹反馈 ←────────────────────┘
                     (高频主题触发新搜索)
```

### 跃迁的本质

| 维度 | 跃迁前 | 跃迁后 |
|------|--------|--------|
| **定位** | 三个独立工具 | 一个生态平台 |
| **数据流** | 单向（采集→存储→查询→结束） | 循环（使用→发酵→进化→更好的使用） |
| **Agent角色** | 被动响应 | 主动协作 + 经验共享 |
| **存储** | 文件夹层级 | 四层仓库 + 跨库关联图 |
| **搜索** | BM25关键词 | 语义 + 关联 + 热值 + 重排序 |
| **学习能力** | 无 | 越用越聪明 |

## 五、系统蓝图：四层架构三层协同

### 5.1 用户界面层

```
Obsidian（知识浏览器） | CLI | REST API | 未来：Web UI
```

### 5.2 Agent-team 层（编排调度）

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  能力注册中心 │→│  动态路由器  │→│   经验池     │
│              │  │              │  │              │
│ Claude: 生信  │  │ 任务→Agent   │  │ "GPT总在这    │
│ GPT: 代码    │  │ 匹配+路由    │  │  类失败"     │
│ 本地: 审查   │  │ 降级回退     │  │ "Claude稳"   │
└──────────────┘  └──────────────┘  └──────────────┘
```

**Agent-team 的三步优化：**

1. **能力注册**：每个 Agent 声明擅长领域和目标成本
2. **动态路由**：根据任务类型、历史表现、成本自动选择最佳 Agent
3. **经验池**：Agent 间的成功/失败模式共享，避免集体踩坑

### 5.3 发酵引擎层

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   触发器    │───→│  规则引擎   │───→│   执行器    │
└─────────────┘    └─────────────┘    └─────────────┘
```

**四条核心发酵规则**：

| 规则 | 触发条件 | 执行动作 |
|------|---------|---------|
| 对话→知识 | 主题T 30天内查询≥3次 | Research-Skill搜索→自动下载→入库→通知 |
| 代码→知识 | 代码模块M复用≥3次 | 关联算法论文→插入引用→生成最佳实践 |
| Agent→Agent | 任务类型T积累≥10样本 | 排名各Agent表现→更新能力画像→优化路由 |
| 错误→模式 | Bug E在多Agent中≥5次 | 提取精确模式→写入规范→前置检查 |

### 5.4 LLM-Wiki 层（四层仓库）

```
知识库 (Chroma)   → 论文、概念、摘要、实体、向量索引、关联权重
对话库 (Markdown) → 历史对话、主题标签、质量评分、情感分析
代码库 (Git)      → 代码片段、设计模式、性能基准、Bug模式
元数据库 (NetworkX) → 跨库关联、Agent画像、热值分布、衰减历史
```

## 六、回答"有什么用"

### 6.1 对个人的价值

| 场景 | 之前 | 之后 |
|------|------|------|
| 查论文 | 手动搜索→自己读 | 语义搜索→自动推荐相关+代码+对话 |
| 写代码 | 从零生成→调试 | 检索已有代码→复用→差异生成 |
| 知识积累 | 存了就忘 | 高频强化→低频衰减→主动推送 |

### 6.2 对实验室的价值

> 实验室有10年的论文，但新人入职3个月才能摸清。FermentAI 让它变成可查询的活知识——问一个问题，系统不仅给答案，还告诉你来自哪篇论文、与哪些代码相关、之前谁讨论过什么。

### 6.3 对企业的价值

> 公司文档散落在各处。FermentAI 统一成图谱。客服 Agent 精准检索，开发 Agent 复用代码，系统还学习哪个 Agent 擅长什么任务。

### 6.4 "不专业的不回答"

> 问：如何做红烧肉？
> 系统检索 → 知识库无相关结果 → 明确告知："该问题超出本实验室知识范围"

vs 通用 LLM：会瞎编做法。

## 七、部署形态

### 个人使用
```
笔记本（8GB RAM）
├── Ollama（本地模型）
├── Chroma（本地向量库）
└── FermentAI API → localhost:8000
```

### 实验室共享
```
服务器（16GB+ RAM，可选GPU）
├── FermentAI API 服务（多用户）
├── 定时发酵任务（自动搜索/更新）
├── 共享 Obsidian Vault（NAS/云同步）
└── 备份
```

### 专业化服务
```
服务器 → REST API 暴露
外部系统 → API 调用 → 获取专业检索结果
成为实验室的"专属领域 AI 服务"
```

## 后记

这个项目起源于一个诚实的问题：**"这有什么用？"**

回答的过程不是堆砌更多技术，而是**让已有的技术流动起来**——让 Research-Skill 的下载数据流入 LLM-Wiki 的知识库，让 LLM-Wiki 的检索结果被 Agent-team 消费，让 Agent-team 的失败经验回流到发酵引擎，驱动新一轮优化。

**核心哲学**：

> 知识不是静态的库存，是动态的过程。
> 数据不是越多越好，是越用越聪明。
> AI 不是被动工具，是主动进化的生态。
